* samplers
 - DEMC is implemented but not wired in (replace with PTDE)
  - Required local patch of PyMC bug
 - Implemented Parallel tempering + Differential Evolution sampler, analogous to EXOFASTv2's sampler
  - Probably better than HMC for multi-modal exploration (like we expect for mulensing)
  - HMC doesn't play well with gradient free Op implementation of MulensModel
  - Robbins-Monro temperature ladder adaptation (Vousden, Farr & Mandel 2016,
    ptemcee): after each swap attempt between rungs k and k+1, update
    Δ(log T_k) += γ_n · (accepted - r*), γ_n ~ 1/n^0.7, r* ≈ 0.234;
    freeze ladder at end of tune phase. Eliminates need to hand-tune T_max.
    ptde.py already has adapt_ladder / target_swap_rate stubs for this.
  - PRIORITY (2026-07-04, see hpc_optimization.txt P12 + vbm_fix.txt §7): a
    real 13,370-step DC2018_128 PTDE run showed per-step eval time jump from
    0.14s to a sustained 1.7-2.0s once the DE population dispersed. Root
    cause: hot tempering rungs routinely draw (s,q,rho) in genuine
    near-caustic territory (11.7% of 870-epoch light curves in that regime
    exceed 1s of VBM contour-integration cost), and with 320 proposals/step
    gathered together, at least one such draw is essentially guaranteed
    every step, gating the whole batch even though T=1 (the only
    scientifically relevant rung) stays cheap. Multi-node MPI (P9) does NOT
    fix this — it's one slow draw, not 320-way queueing pressure. The two
    highest-leverage fixes are: (a) looser VBM Tol/RelTol at hot rungs
    (they're already flattened, don't need 1e-3 photometric precision — this
    directly shrinks the expensive tail), and (b) rung thinning (update hot
    rungs less often, directly reducing the number of chances per step to
    draw from that tail).
    - (b) rung thinning: DONE (2026-07-04) — ptde.py _active_rungs() +
      sampler.rung_thin_factor/rung_thin_start YAML keys. Only reduces the
      FREQUENCY of a hot-rung straggler blocking the batch, not the
      blocking itself; still needs a full-length DC128 run to confirm swap
      rates / T=1 autocorrelation are unaffected (only smoke-tested so far).
    - per-rung timing diagnostic: DONE (2026-07-07) —
      ptde_sample(collect_rung_timing=True) / sampler.collect_rung_timing
      YAML key; logs per-rung n/median/mean/p90/max/n_slow(>0.1s) at the end
      of a run. Result on DC2018_128 (400 steps from MAP): slow calls
      confined to rungs 6-7 only (0/16040 on rungs 0-5; 14/16040 on rung 6;
      115/16040 on rung 7) — concurrency is typically 0-1 stragglers per
      step, not the ~19 previously estimated. See hpc_optimization.txt
      2026-07-07 STATUS UPDATE for the full numbers.
    - (a) per-rung Tol: NOT DONE (implementation). Non-trivial — Tol is
      baked into the magnification Op at graph-compile time, shared by every
      rung's proposals in the one compiled logp function; needs either N
      separate compiled logp functions (one per rung) or a live-mutable Tol
      the worker sets before each _eval_logp call, keyed by which rung a
      proposal belongs to. Correctness-vs-cost question RESOLVED
      (2026-07-07, analytically, see hpc_optimization.txt): the swap-time
      strict-Tol recompute required for detailed balance does NOT negate
      most of the benefit, PROVIDED it is cached per-state (computed once
      per newly-accepted state, reused for subsequent swap attempts on that
      state) rather than recomputed on every swap attempt. Estimated
      overhead ~0.03 s/step against a straggler cost that currently stalls
      ~32% of steps — implement with the cache; skipping it risks negating
      the win for exactly the persistent-straggler case that matters most.
    - also raised: a fully asynchronous/non-blocking dispatch (fast chains
      never wait on a hot-rung straggler) would eliminate the blocking
      entirely rather than just reducing its frequency, but requires
      redesigning how DE proposals draw partner states from
      asynchronously-updating chains and how step-synchronized diagnostics
      are computed per-chain. Not attempted; a bigger design question than
      rung thinning or per-rung Tol. Written up as hpc_optimization.txt P13
      (2026-07-05): NOT a substitute for per-rung Tol — async fixes
      scheduling (nobody idles waiting on a straggler), Tol fixes the
      actual compute cost (hot-rung evaluations are cheaper), and a fixed
      core budget means both matter independently. Do P13 as its own
      scoped session; keep synchronous ptde_sample as the fallback. The
      2026-07-07 diagnostic (above) shows concurrency is typically low
      (0-1 stragglers/step), closer to the original "~26 idle cores" framing
      than the walked-back "~19 concurrent" estimate — informs P13's scope
      if/when it's picked up.

* init_scale: replace symbolic-Jacobian estimation with iterative/empirical probing
  - current approach (config.py relaxation engine) propagates init_scale through
    RELATIONS via symbolic Jacobians -- works but is enormously complex, and only
    encodes how units/relations chain together, not how tightly the data actually
    constrains each parameter
  - proposal: (1) build a first-pass graph using coarse hardcoded/default scales
    just to get a working, finite-loglike model; (2) probe each free element for
    the scale where delta-loglike ~ 1 nat, same mechanism already used to scatter
    PTDE chains (_probe_scales in samplers/ptde.py:92, mirrors EXOFASTv2's
    exofast_getmcmcscale); (3) rebuild the final graph/whitened parameterization
    using those empirical scales as init_scale
  - benefit: grounds init_scale in actual local likelihood curvature (what
    whitening is for) instead of a heuristic proxy; lets us drop the symbolic
    Jacobian solver machinery and the user-facing init_scale override entirely
  - caveats: still need a coarse bootstrap scale to get pass-1 working (not fully
    eliminated, just demoted to "good enough to evaluate"); probe gives marginal/
    diagonal curvature only, so it won't fix mixing from correlated parameters
    (period-t0, e-omega) any better than the current approach does
