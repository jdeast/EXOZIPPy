THE HEADLINE FINDING:
  The bottleneck is NOT VBM contour integration.  dc128.log shows auto_vbbl
  resolved to "hexadecapole everywhere" (max hex-vs-VBM diff 0.0000), and
  MulensModel implements binary-lens hexadecapole as *13 separate python-level
  VBM.BinaryMag0 calls per epoch* through nested python zip loops
  (binarylens.py: BinaryLensHexadecapoleMagnification -> _get_magnification_
  w_plus/_w_times -> _get_1_magnification_point_source).  That is 870 x 13 =
  11,310 pybind round-trips per logp eval.  VBM's BinaryMag2 performs the
  equivalent quadrupole safety test *internally in C++* and falls back to
  point-source when safe — so "VBM everywhere" is simultaneously ~4.3x faster
  AND at least as accurate as the hexadecapole bracketing.  The entire
  auto_vbbl machinery is optimizing for the wrong cost model.

THE HEADLINE FINDING FROM P3's REVIEW (not what was expected):
  VBMicrolensing's BinaryMag2 point-source shortcut requires
  `safedist > 4*rho^2`, but `safedist` (squared distance from the source to
  the caustic region of influence) is only ever COMPUTED for q < 0.01
  (the planetary-caustic formula). For q >= 0.01 it silently keeps its
  initialization value of 10, so for rho > sqrt(10)/2 ~ 1.6 the shortcut can
  NEVER fire — VBM runs the full finite-source contour integration even for
  a source 10^5 Einstein radii from the lens where A=1 to machine precision,
  at ~0.1 s/call. This, not a missing batched API, was the actual cause of
  most of the DC2018_128 eval_timeout rejections (14 of 16 in the original
  log). Fixed in two layers: (a) upstream, in the local VBMicrolensing clone
  (~/python/VBMicrolensing, uncommitted, PR plan in VBMicrolensing.md) —
  safedist = (d - R_inf)^2 for d > R_inf = s + 1/s + 2 when q >= 0.01;
  (b) mirrored in VBMDirectMagOp._magnify (op.py) so the PyPI wheel is fast
  too regardless of whether the upstream PR lands. Validated: caustic
  containment scan (1148 (s,q) geometries, min clearance 9.4% of R_inf),
  boundary-accuracy scan (max |dA| = 5.7e-5 at the guard boundary, 0.06x
  VBM's Tol), 4000-point random A/B (max rel diff 1.5e-4). See vbm_fix.txt
  sections 1-4 for the full derivation and section 5 for limitations.

PRODUCTION RESULT — the ~10x step speedup did NOT materialize in aggregate,
and the reason reprioritizes everything below (full data in vbm_fix.txt §7,
reproducible with scripts/repro_vbm_timeout.py --epochs on
examples/DC2018_128/fitresults/DC2018_128.log):
  A real 13,370-step PTDE run (2026-07-04, vbm_direct backend + the
  far-field guard) shows per-step eval time start at 0.14 s (matches the P1+P2
  expectation) then jump to a SUSTAINED 1.7-2.0 s for essentially the entire
  rest of the 7-hour run — ~10x WORSE than the pre-fix baseline, not better.
  eval_timeout events dropped from 16 (in ~1700 steps) to 3 (in 13,370 steps)
  — the far-field bug genuinely is fixed — but replaying those 3 events in
  isolation (scripts/repro_vbm_timeout.py --epochs) shows they complete in
  2-3 MILLISECONDS: their own parameter values were never intrinsically slow.
  The sustained slowdown is instead the NEAR-ZONE finite-source cost
  distribution (genuine caustic-adjacent contour integration, which the
  far-field guard deliberately does not touch) sampled at hot tempering
  rungs: characterizing 870-epoch light curves at (s,q,rho) drawn uniformly
  from the prior a hot rung explores shows a heavy tail — median 15 ms, but
  P90 1.7 s, 11.7% of curves exceed 1 s. With 320 proposals/step and the
  sampler's gather blocking on the SLOWEST of all 320, the odds that NONE
  land in that tail are (1-0.117)^320 ~ 1e-17 — i.e. every step is
  essentially guaranteed to contain one genuinely expensive draw, almost
  always from a hot rung (T=1's posterior stays tight near the real
  solution and is cheap). This is a TEMPERATURE-LADDER / SCHEDULING problem,
  not a magnification-backend bug.

  REPRIORITIZATION: P9 (multi-node MPI) does NOT fix this — more cores only
  help when MANY of the 320 proposals are slow at once (queueing/dispatch
  latency); here it is typically ONE slow draw per step, and that single
  worker's task duration is unaffected by how many other cores exist. P12's
  ideas (rung thinning, and a new one below: looser Tol at hot rungs) attack
  the actual mechanism and are now the priority. P9 is demoted to "do once
  n_params grows enough that dispatch latency dominates," not before.

  - The parallax offsets delta_N(t), delta_E(t) are INDEPENDENT of the

(Roman data challenges, eventually the mission) is thousands of independent
events — embarrassingly parallel at the event level.  A cluster is better
spent running N events at 64 cores each than one event at N*64.
Task:

   - a manifest file (one yaml path per line), SGE_TASK_ID indexes into it;
   - per-event working dir + prefix derived from the yaml "prefix" key;
   - per-event `maxtime` guard and clean early-stop (PTDE already supports

PROMPT 12 — PTDE algorithmic experiments (PROMOTED — this is the fix for the
2026-07-04 production finding: a hot-rung near-zone finite-source draw gates
essentially every step.  Items 0 and 1 directly attack that; do them first.)
--------------------------------------------------------------------------------
0. (NEW, highest priority) Looser Tol/RelTol at hot rungs. Hot rungs (large T)
   already evaluate a heavily flattened target (lp/T) — they don't need
   1e-3 photometric precision, they need to keep the DE population diverse
   enough to feed swaps. VBMDirectMagOp / BinaryLensMagOp currently share one
   Tol across all rungs. Give the Op a per-rung Tol (e.g. Tol=1e-3 at k=0,
   relaxing to ~1e-2 or ~3e-2 by k=K-1); BinaryMagDark's annuli loop exits
   as soon as currerr < Tol, so this directly shrinks the expensive tail
   (scripts/ nearzone_cost.py-style benchmarking showed median 15ms/mean
   445ms/P90 1.7s at Tol=1e-3 for near-zone light curves — rerun that sweep
   at Tol=1e-2/3e-2 to quantify the win before implementing). Validate: swap
   acceptance rates and T=1 posterior unchanged (only hot-rung *precision*
   changes, not their target distribution's support).
1. Rung thinning: hotter rungs exist only to feed swaps; try updating rungs
   k>=K/2 every 2nd step (halves eval count ~25-40%, and halves the number
   of chances per step for a hot rung to land in the expensive tail).
   Validate: swap acceptance rates and T=1 autocorrelation time unchanged on
   DC128.
2. Adaptive temperature ladder: the Robbins-Monro hook (target_swap_rate,
   adapt_ladder) is already reserved in the API (ptde.py line ~254).
   Implement Vousden et al. 2016-style ladder adaptation during tune,
   targeting uniform swap acceptance ~0.2-0.4; often lets n_temps drop from
   8 to 4-6 => proportional eval savings.
3. Snooker DE moves (ter Braak & Vrugt 2008) mixed at ~10%: better mixing at
   fixed n_chains, possibly allowing n_chains < 2*n_params for large models.
4. Convergence-informed draw budget: with min_ess=1000 the configured
   draws=252600 is a ceiling, not a target — after the speedups, consider
   defaulting examples to a large ceiling + convergence stop (already the
   mechanism) and document that pattern in the README.
Each experiment: implement behind a flag, compare ESS/second against baseline
on DC128 with identical seeds, keep only what wins.
