THE VBMicrolensing "safedist" FIX — what it does, why it is safe, and its limits
================================================================================
Date: 2026-07-04.  Author: Claude (Fable 5), reviewed numerics on radish.
Applies to: VBMicrolensing <= 5.5 (github.com/valboz/VBMicrolensing).
Local patch: ~/python/VBMicrolensing/VBMicrolensing/lib/VBMicrolensingLibrary.cpp,
  BinaryMag0(), ~line 765 (uncommitted; PR plan in VBMicrolensing.md).
EXOZIPPy mirror guard: src/exozippy/components/mulensing/op.py,
  VBMDirectMagOp._magnify (active regardless of which VBM wheel is installed).
Validation script: scripts/validate_vbm_safedist.py (reproduces Scans 1 & 2).

--------------------------------------------------------------------------------
1. EXECUTIVE SUMMARY
--------------------------------------------------------------------------------
VBMicrolensing's BinaryMag2 decides per call whether the finite-source
computation is needed or the point-source value suffices.  One of its three
safety tests uses `safedist`, the squared distance of the source from the
"caustic region of influence" — but safedist is only actually computed for
q < 0.01 (a planetary-caustic formula).  For q >= 0.01 it silently keeps its
initialization value of 10.  The test is `safedist > 4*rho^2`, so for
q >= 0.01 and rho > sqrt(10)/2 ~ 1.6 the point-source shortcut can NEVER
fire: the full finite-source machinery runs even for a source 10^5 Einstein
radii from the lens where A = 1 to machine precision, at ~0.1 s per call —
~10^4 x the point-source cost, for zero information.

The fix supplies the missing geometry for q >= 0.01: every caustic of a
binary lens lies inside a circle of radius R_inf = s + 1/s + 2 around the
center of mass, so for source distance d > R_inf we set
safedist = (d - R_inf)^2.  Nothing else changes; in particular the two
quantitative accuracy tests (corrquad, corrquad2) still gate the shortcut.

Measured: a pathological 870-epoch light curve (s=0.12, q=2.1, rho=10.8,
source at u ~ 3000–6000) went from 55.7 s to 0.119 s with results unchanged.
This was the root cause of 14 of the 16 eval_timeout rejections in the
DC2018_128 PTDE run (hot tempering rungs propose exactly this kind of junk;
each rejected logp then cost a 10 s stall plus a worker-pool recycle).

--------------------------------------------------------------------------------
2. HOW BinaryMag2 DECIDES (upstream logic, unchanged by the fix)
--------------------------------------------------------------------------------
BinaryMag2(s, q, y1, y2, rho):
  1. Computes the point-source magnification Mag0 = BinaryMag0(...), which as
     side products evaluates, from the actual image configuration:
       corrquad  — an estimate of the quadrupole (finite-source) correction
                   built from the Jacobian derivatives of the real images;
       corrquad2 — a ghost-image/cusp proximity diagnostic built from the
                   would-be image pair when only 3 real images exist;
       safedist  — squared distance of the source from the caustic region
                   of influence (geometry, not image derivatives).
  2. Applies the test
       corrquad*6*(rho^2+..) < Tol  AND  corrquad2*256*(rho^2+..) < 1
       AND  safedist > 4*rho^2
     If it passes: return Mag0 (point source).  Else: run the full
     limb-darkened contour integration (BinaryMagDark -> annuli ->
     BinaryMagSafe -> BinaryMag).

Why three tests?  corrquad/corrquad2 are evaluated AT THE SOURCE CENTER.
A small quadrupole correction at the center does not exclude that the
source LIMB overlaps a caustic elsewhere (the finite-source deviation is
then O(1), not O(rho^2)).  safedist closes exactly that blind spot: it
requires the nearest caustic structure to be farther than 2*rho from the
source center, i.e. at least rho beyond the limb.  The 2*rho margin is
upstream's own threshold (used for years by the q < 0.01 planetary branch);
the fix does not touch it.

The bug: for q >= 0.01, safedist is never computed.  It keeps the
initialization `safedist = 10`, which acts as "the caustic region extends to
sqrt(10) ~ 3.2 Einstein radii from wherever the source is" — a constant,
independent of the actual source position.  Consequence: rho > 1.6 makes
`10 > 4 rho^2` false for every epoch of every light curve, no matter how far
the source is from the lens.

--------------------------------------------------------------------------------
3. THE FIX
--------------------------------------------------------------------------------
In BinaryMag0, after the existing q < 0.01 planetary branch:

    else {
        static double sabs, Rinf, d2;
        sabs = fabs(a.re);                    // |s|; a.re sign depends on q<1
        Rinf = sabs + 1.0 / sabs + 2.0;
        d2 = y1v * y1v + y2v * y2v;           // source distance^2 from CoM
        if (d2 > Rinf * Rinf) {
            safedist = d2 + Rinf * Rinf - 2 * sqrt(d2) * Rinf;   // (d - Rinf)^2
        }                                     // else: keep 10 (= upstream)
    }

Semantics:
  - d <= R_inf  : safedist stays 10 — bit-for-bit upstream behavior.
  - d >  R_inf  : safedist = (distance to the guard circle)^2, a strict
                  LOWER bound on the squared distance to any caustic.
The shortcut therefore fires when the source center is farther than
R_inf + 2*rho from the center of mass (given the corrquad tests also pass).

The EXOZIPPy-side mirror (VBMDirectMagOp._magnify) applies the identical
criterion in numpy and dispatches far epochs to BinaryMag0/MultiMag0
directly, so runs against unpatched PyPI wheels get the same benefit, and
the N-lens path (MultiMag2 has the same safedist=10 issue upstream) is
covered with R_inf = max_j(s_j + 1/s_j) + 2 over companions.

--------------------------------------------------------------------------------
4. WHY IT IS SCIENTIFICALLY SAFE (the referee section)
--------------------------------------------------------------------------------
(a) The fix relaxes only the geometric veto.  The two quantitative accuracy
    tests — corrquad (actual quadrupole correction from the real images) and
    corrquad2 (ghost-image proximity) — are still required for the shortcut.
    A source whose center shows any appreciable finite-source correction
    still takes the full path, exactly as before.

(b) Caustic containment (Scan 1 of scripts/validate_vbm_safedist.py).
    Claim: all caustics lie within R_inf = s + 1/s + 2 of the CoM.
      - Wide binaries (s > 1): the two caustics sit near each body; the
        farther body is at most s from the CoM and its caustic is
        sub-Einstein in size  =>  extent <~ s + 1.
      - Close binaries (s < 1): the secondary (triangular) caustics lie at
        distance ~ 2 sqrt(q) / (s (1+q)) from the CoM, which approaches 1/s
        for q -> 1.  THE 1/s TERM IS LOAD-BEARING, not padding.
      - The +2 supplies the safety margin on top.
    Numerical check: VBM.Caustics on a 41 x 28 grid covering
    s in [0.05, 20], q in [1e-9, 1] (VBM internally maps q>1 to 1/q, so this
    covers all mass ratios): 1148 geometries, max caustic radius = 0.906 R_inf
    (at s=0.05, q=1, r_caustic=19.98 vs R_inf=22.05).  Caustics never reach
    the guard circle; minimum clearance 9.4% of R_inf.  By s-regime:
        close    (s<0.5)      max ratio 0.906
        resonant (0.5<=s<=2)  max ratio 0.361
        wide     (s>2)        max ratio 0.905

(c) Photometric accuracy ON the guard boundary (Scan 2).
    Worst admissible configuration: source center exactly at
    d = R_inf + 2*rho, strongest limb darkening a1 = 1, compared against the
    full finite-source computation (BinaryMagDark, Tol=1e-5, which never
    shortcuts).  Grid: s in {0.1..10}, q in {1e-6..1}, rho in {0.01..10},
    12 position angles = 1500 configurations:
        max |A_ps - A_fs| = 5.7e-5      (0.057 x VBM's default Tol = 1e-3)
    i.e. ~0.06 mmag at baseline — an order of magnitude below Roman
    photometric noise, and the error decays steeply (the finite-source
    correction falls off faster than d^-4) beyond the boundary.  Note the
    worst case occurs where the total signal itself is only A = 1.0014.

(d) Distributional A/B: 4000 random (s, q, x, y, rho, a1) draws spanning
    s in [0.05, 20], q in [1e-6, 1e2], rho in [3e-4, 16], d in [0.01, 30]:
    max relative difference patched vs unpatched = 1.5e-4, zero points above
    Tol = 1e-3, all differences localized at the shortcut boundary where
    both answers are within tolerance of truth by construction.

(e) End-to-end: DC2018_128 starting lp identical to 5e-12 between the
    vbm_direct (guarded) and mulensmodel backends; full EXOZIPPy test suite
    (305 tests, incl. tests/test_vbm_direct_vs_mulensmodel.py which spans
    the guard boundary against MulensModel) passes; a 625-step PTDE run
    produced ZERO eval_timeout events (the original run averaged ~1 per
    100 steps).

Interpretation for planet discovery: the shortcut only ever replaces the
finite-source calculation in a regime where (i) the source limb is provably
at least rho from any caustic (containment scan), (ii) the center shows no
measurable quadrupole correction (corrquad tests), and (iii) the worst-case
residual is 5.7e-5 in magnification.  A planetary signal, by definition,
comes from the source interacting with caustic structure — which is inside
the guard circle, where the fix changes nothing at all.

--------------------------------------------------------------------------------
5. LIMITATIONS AND CAVEATS (document these with any published use)
--------------------------------------------------------------------------------
1. Scope: the patch touches only the q >= 0.01 branch of the BINARY
   safedist.  The planetary branch (q < 0.01) is untouched.  The multi-lens
   MultiMag2 path has the same upstream defect and is NOT patched in VBM
   (kept out for PR minimality) — EXOZIPPy's op-level guard covers it.
2. Inside the guard circle nothing improves: a q >= 0.01, rho > 1.6 source
   NEAR the lenses still takes the expensive path.  That computation is
   CORRECT, just slow; in EXOZIPPy it is mitigated by the rho <= 10 soft
   barrier (defaults.yaml) and the PTDE eval_timeout.
3. The 2*rho safety margin is inherited from upstream, not re-derived.  Our
   boundary scan (4c) empirically validates it at <= 5.7e-5 for the grids
   above; rho > 10 and s outside [0.05, 20] were not scanned (for larger s,
   R_inf grows, so the guard simply fires farther out — the conservative
   direction).
4. Astrometry: when the shortcut fires, astrox1/astrox2 are the
   point-source centroid.  This is upstream's existing shortcut semantics,
   not a new approximation — but if microlensing astrometry at the
   sub-rho level far from caustics ever matters, revisit.
5. Validation used VBM's own Caustics/BinaryMagDark as truth.  Both are
   independent code paths from the shortcut logic, but they share the
   underlying lens-equation solver (an error in NewImages would affect both
   sides of the comparison).  The MulensModel cross-check (4e) is the
   independent-implementation guard.
6. Known upstream oddities noticed but deliberately NOT fixed here (PR
   minimality; each is a candidate follow-up):
     - BinaryMagSafe's "weird" heuristic `Mag*RS > 3` is unconditionally
       true for RS > 3 (Mag >= 1 always), triggering its most expensive
       retry cascade precisely for giant sources.  `(Mag-1)*RS > 3` would
       preserve the small-RS intent without that failure mode.
     - BinaryMagSafe's second retry loop tests `mag2 * RSi > 3` where the
       loop variable is RSo — almost certainly a typo.
7. Deployment status (as of 2026-07-04): the patch lives ONLY in the local
   clone (~/python/VBMicrolensing, uncommitted, alongside the GIL-release
   patch).  The poetry env still installs the PyPI 5.5 wheel; production
   EXOZIPPy runs are protected by the op-level mirror guard, not by the
   patch.  Building the local copy requires
   CXX=/opt/rh/gcc-toolset-13/root/usr/bin/g++ (system gcc 8.5 < 9).

6. RECOMMENDED PRIOR BOUNDS (defaults.yaml) UNTIL THE PR LANDS — AND AFTER
--------------------------------------------------------------------------------
Even with the op-level guard, two slow-but-correct regimes remain reachable:
sources INSIDE the guard circle with large rho (regime of caveat 5.2), and
extreme-magnification wide binaries (s ~ 20, q ~ 1e-6, u0 ~ rho: every epoch
is a genuine 0.1–1 s finite-source computation; observed as the two
"legitimate" DC128 timeout sets).  Bounds that close these off without
touching any conceivable discovery:

  s: [0.1, 10]        (was [0.05, 20])
     - Every published binary/planetary microlensing solution has
       0.1 < s < 6 (the old defaults comment itself noted smallest ~0.3,
       widest ~5.3); [0.1, 10] leaves >~ 2x margin beyond all of them.
     - Detectability: caustic sizes scale as ~ q s^2 (close) and ~ q / s^2
       (wide); beyond s ~ 10 a companion needs q >~ 0.1 to leave a
       detectable signature, and such a system is better modeled with the
       near body as the lens (a genuinely wide massive companion acts as
       external shear).  If a specific event really needs s > 10, override
       the bound in that event's params file — bounds in defaults.yaml are
       defaults, not physics.
     - Performance: halves R_inf (22 -> 12.1), shrinking the expensive
       near-zone AREA ~4x, and removes the s=20 prior wall where 5 of the
       16 DC128 timeout excursions piled up (including both
       extreme-magnification sets).
  q: [1e-8, 100]      (was [1e-10, 100])
     - Lower: the most optimistic Roman detection floor is ~Moon mass
       (q ~ 1e-7 for a 0.3 Msun lens); 1e-8 sits 10x below it.  Below that
       the companion is pure nuisance volume — likelihood exactly flat in
       (s, alpha, q), which parallel tempering will happily waste steps in.
     - Upper 100 unchanged: q > 1 is legal and used (e.g. ob161003's
       stellar companion); VBM symmetrizes internally, cost is q-symmetric.
  rho: upper 10       (already applied; see defaults.yaml comment)
     - rho >~ a few both erases the signal (A -> 1 + 2/rho^2) and is the
       most expensive VBM regime when combined with q >= 0.01 near-zone
       trajectories.  No lower bound: rho > 0 by construction and a barrier
       at 0 would penalize real small-rho fits.

Note these bounds are the DEFENSE IN DEPTH layer: the far-field pathology
itself is already fixed by the guard/patch; the bounds keep tempering chains
out of the residual slow-but-correct corners, and they also apply to the
mulensmodel A/B backend, which has no op-level guard.

--------------------------------------------------------------------------------
7. PRODUCTION RESULT: did the far-field fix deliver the ~10x step speedup?
--------------------------------------------------------------------------------
NO — not in aggregate wall-clock time over a full run, and the reason is
instructive: fixing the far-field pathology unmasked a different, more
COMMON bottleneck that was always there but was previously smaller than the
far-field disaster.

Evidence (examples/DC2018_128/fitresults/DC2018_128.log, a real 13,370-step
PTDE run, 2026-07-04 13:36-20:45, using the fixed vbm_direct backend +
far-field guard):
  - eval_timeout=10s events: only 3 across the whole run (vs 16 in ~1700
    steps pre-fix) -- the far-field pathology IS fixed, by a large margin.
  - Per-step eval time (scripts/repro_vbm_timeout.py + log parsing):
      steps      0- 100: median 0.139s  (matches the early P1+P2 expectation)
      steps    100- 500: median 0.158s, mean climbing (0.667s) -- outliers appearing
      steps   500-1500: median 1.820s  -- the new floor, and it never comes back down
      steps  1500-13370: median stays 1.7-2.0s for the REST of the 7-hour run
    i.e. once the DE population dispersed (chain init starts tight, disperses
    over the first ~500-1500 steps -- see PTDE ensemble-start comments), the
    STEADY-STATE per-step cost ended up ~10x WORSE than the pre-fix baseline
    (~0.18-0.20s/step), not ~4x better.
  - Reproducing the 3 explicit timeout events in isolation (single process,
    no contention) via scripts/repro_vbm_timeout.py --epochs: all 3 replay
    in 2-3 MILLISECONDS, not 10+ seconds. Their own parameter values were
    never intrinsically slow -- rho was small (9e-4 to 6e-3) and q>1 (5.2 to
    28.6) placed them squarely inside VBM's existing q<0.01 planetary-branch
    safedist (irrelevant here) but also inside our general-q guard's near
    zone at these s (8.3-9.9, within [0.1,10]). The logged timeout must
    reflect transient conditions at the moment (worker-pool contention from
    OTHER concurrently-dispatched proposals sharing the box), not a cost
    inherent to the logged parameters -- scripts/repro_vbm_timeout.py prints
    this NOTE automatically when a replay is much faster than the logged
    timeout.
  - Root cause of the SUSTAINED (not just the 3 rare) slowdown: characterized
    directly (scripts/ nearzone_cost.py during development; not checked in,
    reproduce with the numbers below). Sampling (s,q,rho) uniformly across
    the prior support a hot rung would explore (s in [0.1,10], q in
    [1e-8,100], rho in [1e-4,10]) with near-zone trajectory positions (inside
    R_inf, i.e. genuinely requiring the contour integration -- the regime
    our far-field guard does NOT touch by design):
      - single-call cost: median 4 us, mean 0.8 ms -- most calls are still
        cheap (point-source-safe within the near zone too).
      - but a FULL 870-epoch light curve at FIXED (s,q,rho) sweeping the
        trajectory: median 15 ms, MEAN 445 ms, P90 1.7s, MAX 6.2s.
        11.7% of curves exceed 1s; 8.3% exceed 2s.
    With 320 proposals/step (8 rungs x 40 chains) and the sampler's gather
    blocking until the SLOWEST of all 320 finishes, the probability that
    NONE of 320 draws lands in the >1s tail is (1-0.117)^320 ~ 10^-17 --
    i.e. every single step is expected to contain at least one genuinely
    expensive near-zone light curve once the population has dispersed
    across the prior. That single slow draw (routinely from a HOT,
    scientifically uninteresting rung -- T=1's posterior stays tightly
    constrained near the real solution, s~0.98 q~0.0011 rho~0.005, which is
    fast) gates the entire step.

Conclusion: our optimizations (P1 hexadecapole removal, P2 MulensModel
bypass, the far-field guard) all work exactly as designed and were verified
independently (see sections 1-4 and tests/test_vbm_direct_vs_mulensmodel.py).
But the ~10x aggregate speedup estimated in hpc_optimization.txt assumed the
evaluated points would mostly resemble the well-behaved MAP; a parallel-
tempering ladder does the opposite by construction -- it deliberately
samples badly-behaved (compute-cost-wise) regions at high T. Once the far-
field bug (which used to dominate and mask this) was fixed, the true
long-pole became "at T>1, a nontrivial fraction of proposals require
genuine, correctly-computed, but expensive finite-source contour
integration, and a per-step batched gather over 320 proposals means one
such draw taxes everyone." This is fundamentally a TEMPERATURE-LADDER /
SCHEDULING problem, not a magnification-backend bug -- exactly what
hpc_optimization.txt's P12 (rung thinning, adaptive ladder targeting
uniform swap acceptance, snooker DE) and P9 (per-rung-independent dispatch)
already anticipated as the next tier of work. Concretely worth trying next:
  - Rung thinning: update rungs k >= K/2 every 2nd (or 4th) step -- directly
    cuts the number of chances per step for a hot-rung outlier to gate the
    batch.
  - Cost-aware/adaptive Tol per rung: hot rungs need much less magnification
    precision than T=1 (they're already flattened/near-prior); a looser Tol
    (e.g. 1e-2) for k>=K/2 would shrink the contour-integration tail directly.
  - Tighter rho/s bounds (section 6) reduce the volume of the expensive
    near-zone tail somewhat but cannot eliminate it -- genuine caustic
    interaction at realistic rho is unavoidably expensive and is exactly the
    signal we're trying to detect at T=1.
